## 《洪武立志传》核心数据层搭建

核心数据层是游戏运行的“唯一真相源”，所有系统模块（合战、个人战、主命、社交、经济等）均读取并更新这一层的数据。设计得当的核心数据层，可以让你在单人开发中做到“改一处而全局同步”，极大降低维护成本。

以下从**设计原则**、**数据表结构**、**JSON示例**、**初始化流程**四个层面展开。


### 一、设计原则

1. **卡片驱动**：所有可收集内容（人物、技能、战术、称号、名所）统一为“卡片”结构，以`card_id`作为全局唯一标识。系统间的奖励、解锁均通过授予`card_id`完成。

2. **最小冗余**：能通过ID引用的绝不复制完整数据。例如`owned_cards`只存卡片ID列表，详细定义在独立卡片库中。

3. **状态与定义分离**：静态数据（如技能上限、卡片效果）与动态数据（如玩家当前技能等级、城镇防御度）分表存储。静态数据可作为游戏内置常量，动态数据存于存档。

4. **版本兼容**：为每个数据表预留`version`字段，便于后续扩展时做存档迁移。

5. **可读性优先**：字段命名采用中文拼音或英文均可，但需统一；注释详尽，方便隔周后自己仍能看懂。


### 二、数据表结构总览

```
┌────────────────────────────────────────────────────────────────┐
│                        核心数据层                               │
├────────────────────────────────────────────────────────────────┤
│  【静态库】（游戏内置，不随存档改变）                             │
│  ├── 角色模板库（char_templates）     所有NPC的基础属性/技能      │
│  ├── 卡片库（card_library）           600+卡片定义               │
│  ├── 城镇定义库（city_defs）          各城镇初始状态              │
│  ├── 势力定义库（faction_defs）       历史势力初始数据            │
│  ├── 技能定义库（skill_defs）         16项技能及效果              │
│  ├── 主命模板库（mission_templates）  所有主命的参数              │
│  └── 事件库（event_library）          剧情触发条件与对话           │
│                                                                  │
│  【动态存档】（玩家存档文件）                                     │
│  ├── 玩家角色数据（player）            当前能力/技能/卡片/称号     │
│  ├── NPC实例数据（npcs）              已生成NPC的当前状态          │
│  ├── 势力实例数据（factions）         当前各方势力状态             │
│  ├── 城镇实例数据（cities）           各城镇当前属性               │
│  ├── 关系网络（relationships）        角色间亲密度                 │
│  ├── 全局时间（global_time）          年/月/日                    │
│  └── 事件标记（event_flags）          已触发事件、开关             │
└────────────────────────────────────────────────────────────────┘
```

#### 4.5 关系网络（`relationships.json`）

以邻接表形式存储角色间的亲密度。只存储非默认值（默认0）。

```json
{
  "relationships": {
    "ZHU_YUANZHANG_XU_DA": 4,
    "ZHU_YUANZHANG_CHANG_YUCHUN": 3,
    "ZHU_YUANZHANG_LIU_BOWEN": 3,
    "ZHU_YUANZHANG_MA_XIUYING": 5,
    "ZHU_YUANZHANG_CHEN_YOULIANG": -3
  }
}
```

亲密度范围：-4（死敌）至 5（生死相随）。0为中立。正数每颗心对应1级，4心为最高非配偶关系。

### 六、初始化流程（游戏启动/新游戏）

1. **加载静态库**：将所有`_defs.json`和`_library.json`读入内存，作为全局常量。

2. **创建新存档**（以第一章为例）：
   - 从`char_templates`读取`ZHU_YUANZHANG`模板，赋值给`player`
   - 设置初始时间：1328年（或直接从1353年投军开始）
   - 初始化必要NPC（郭子兴、马秀英、汤和等）到`npcs`
   - 根据`faction_defs`初始化`factions`和`cities`
   - 设置初始`event_flags`
   - 授予玩家初始卡片（如“皇觉寺”名所卡）

3. **存档/读档**：
   - 存档：将当前内存中的动态数据（player, npcs, factions, cities, relationships, world_state）序列化为JSON文件
   - 读档：读取JSON文件，反序列化覆盖当前动态数据；静态库始终从原始文件加载

4. **数据校验与迁移**：
   - 读档时检查存档版本，若低于当前游戏版本，执行迁移脚本（如添加新字段、修正卡片ID等）


### 七、扩展性设计建议

- **卡片ID命名规范**：`类型_子类_序号`，如`CHAR_XUDA`、`TACTIC_FIRE_001`、`TITLE_STRATEGIST`。便于程序批量处理和人工调试。
- **预留字段**：每个数据对象建议加`ext`字段（空对象），用于未来扩展而不破坏结构。
- **事件驱动更新**：当某个系统修改了核心数据（如合战胜利后城池易主），应通过事件通知相关系统刷新界面，而非各系统轮询检查。
- **单人开发简化**：若时间有限，可先以单一大JSON文件管理全部数据，后期再拆分。关键保持结构清晰、注释完整。


### 八、本章小结

核心数据层是《洪武立志传》的基石。通过将“静态定义”与“动态状态”分离，你可以在不修改存档结构的情况下持续增加新卡片、新任务。各系统模块只需读取/更新这些数据，彼此解耦，便于独立开发和调试。

有了这套数据层，下一步你可以：
1. 编写一个简单的Python脚本，从JSON读取数据，在控制台打印玩家状态
2. 实现最基础的“移动”和“时间推进”
3. 逐步接入评定会、任务、战斗系统

数据层搭稳了，后面的功能就像拼积木一样顺畅。
