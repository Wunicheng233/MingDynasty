## 《洪武立志传》剧情与事件系统完整设计

剧情与事件系统是《洪武立志传》的**叙事引擎**与**世界驱动器**。它将历史洪流与个人奋斗编织成一张绵密的网——玩家既会亲历“鄱阳湖火攻”“北伐中原”等决定天下命运的宏大叙事，也会遭遇“路见不平”“商帮纠纷”等随机偶发的人生插曲。事件系统的核心使命是：**让玩家感觉自己正置身于一个鲜活的明初世界，每一步选择都在书写属于自己的历史。**

以下从事件类型、触发机制、事件链设计、与核心系统的耦合、数据结构、文字界面实现以及明朝特色事件案例七个层面展开。


### 一、事件类型与层级

游戏中的事件按重要性与触发方式分为四大类：

| 类型 | 说明 | 触发方式 | 是否可重复 | 典型示例 |
|------|------|---------|-----------|---------|
| **主线事件** | 驱动第一章历史进程的必发事件，构成朱元璋从乞丐到皇帝的核心叙事 | 时间+条件自动触发 | 否（单存档内） | 投身红巾军、娶马秀英、鄱阳湖决战、登基称帝 |
| **支线事件** | 与特定角色、地点、身份相关的剧情，丰富世界但不影响主线推进 | 条件触发（可错过） | 部分可重复 | 寻访刘伯温、武当学艺、郑和宝船传闻 |
| **随机事件** | 移动或停留时概率触发的小插曲，增加世界生动感 | 概率触发 | 可重复 | 遭遇山贼、行商叫卖、路遇难民、天象异变 |
| **定期事件** | 固定在每年/每月特定日期发生的事件 | 时间自动触发 | 每年/每月重复 | 评定会、新年朝贺、科举考试、武举乡试、秋收祭天 |


### 二、事件触发机制

事件系统采用 **“条件检测 + 优先级队列”** 的模式运行。游戏每次推进时间（以“天”为单位），都会检测当前是否有满足条件的事件待触发。

#### 2.1 触发条件类型

| 条件类别 | 具体条件示例 | 数据来源 |
|---------|-------------|---------|
| **时间条件** | 当前日期 ≥ 1356年3月1日；当前为奇数月1日 | `global_time` |
| **地点条件** | 玩家位于应天府、凤阳府等特定城镇 | `player.location` |
| **身份条件** | 玩家身份为“红巾军士兵”或“吴王” | `player.identity` |
| **势力条件** | 玩家所属势力为朱元璋军；某势力已灭亡 | `player.faction`，`factions` |
| **亲密度条件** | 与徐达亲密度 ≥ 3心；与马秀英亲密度 ≥ 4心 | `relationships` |
| **持有卡片条件** | 持有“刘伯温”人物卡；持有“火攻”战术卡 | `player.owned_cards` |
| **技能条件** | 武艺技能 ≥ Lv2；兵法技能 ≥ Lv3 | `player.skills` |
| **事件标记条件** | 事件“郭子兴病逝”已触发；标记“已娶马氏”为真 | `event_flags` |
| **随机概率** | 移动时10%概率触发 | 随机数 |

#### 2.2 事件优先级与冲突处理

当多个事件同时满足条件时，按以下优先级触发：
**主线事件 > 支线事件 > 定期事件 > 随机事件**

同优先级内，按事件ID排序。每个事件触发后会设置“已触发”标记，防止重复触发（随机事件除外）。

#### 2.3 事件触发流程图

```
时间推进（advance_time）
        │
        ▼
遍历所有未触发事件，检测条件
        │
        ▼
    ┌───────────────┐
    │ 有满足条件？   │
    └───────┬───────┘
            │ 是
            ▼
    ┌───────────────┐
    │ 按优先级排序   │
    └───────┬───────┘
            │
            ▼
    ┌───────────────┐
    │ 执行事件逻辑   │
    │ · 显示剧情文本 │
    │ · 玩家选择分支 │
    │ · 更新世界状态 │
    │ · 授予卡片/奖励│
    │ · 设置事件标记 │
    └───────┬───────┘
            │
            ▼
    ┌───────────────┐
    │ 继续时间推进   │
    └───────────────┘
```

主线事件构成一条严密的因果链，推动朱元璋的人生阶段跃迁。以下是第一章核心事件链的简化图谱：

```

  1344年 · 皇觉寺遣散，游方乞讨（触发游方僧生涯，解锁移动能力）

【阶段二：投身义军】
  1353年 · 收到汤和来信，决定投奔郭子兴（时间+地点条件）

  1353年 · 郭子兴收为亲兵，赐名“元璋”
```

### 四、事件与核心系统的耦合

事件系统是“胶水层”，它将各独立系统串联成有机整体。

| 关联系统 | 事件如何影响该系统 | 系统如何影响事件触发 |
|---------|-------------------|---------------------|
| **时间系统** | 事件推进时间（如“数月后”跳跃）；事件触发依赖时间 | 时间到达是事件的首要条件 |
| **合战系统** | 历史战役事件直接启动合战，并预设特殊参数（如火攻伤害倍率） | 合战胜利/失败设置事件标记，影响后续事件分支 |
| **个人战系统** | 剧情单挑（如朱元璋单挑陈友谅部将）触发个人战 | 个人战胜负影响剧情分支 |
| **卡片系统** | 事件奖励人物卡、战术卡、事件卡、称号卡 | 持有特定卡片是触发某些事件的前提（如持“刘伯温”卡才能触发其献策事件） |
| **主命系统** | 事件可能给予临时主命（如“护送粮草至洪都”） | 完成主命设置事件标记 |
| **社交系统** | 事件大幅改变亲密度（如救命之恩+40）；婚姻事件锁定亲密度 | 亲密度达标触发结义/婚姻事件 |
| **经济系统** | 事件奖励金钱、宝物；消耗金钱（如赈灾事件） | 资产数量可能影响事件选项（如“捐钱修庙”需金钱≥100贯） |
| **世界状态** | 事件改变势力版图（如郭子兴死后势力分裂）、城镇归属、角色生死 | 世界状态是事件条件的重要组成部分 |


### 五、事件数据结构设计（JSON）

事件库作为静态数据，每条事件记录包含触发条件、事件内容、选项分支、效果四部分。

```json
{
  "event_id": "E_MAIN_ZHU_TOUJUN",
  "type": "main",
  "name": "投身红巾",
  "trigger": {
    "time": { "year": 1353, "month": 3, "day": 1, "operator": ">=" },
    "location": "HAOZHOU",
    "identity": "游方僧",
    "flags_required": [],
    "flags_forbidden": ["E_MAIN_ZHU_TOUJUN_DONE"],
    "probability": 1.0
  },
  "scenes": [
    {
      "scene_id": 1,
      "text": "濠州城外，红巾军大营。你望着营门前飘扬的旗帜，想起汤和信中的话：‘与其饿死沟壑，不如搏个封妻荫子！’……",
      "choices": [
        {
          "text": "大步走入军营，投效郭子兴！",
          "effects": [
            { "type": "change_identity", "value": "红巾军士兵" },
            { "type": "change_faction", "value": "郭子兴军" },
            { "type": "add_card", "card_id": "CHAR_GUO_ZIXING" },
            { "type": "set_flag", "flag": "E_MAIN_ZHU_TOUJUN_DONE" }
          ],
          "next_scene": 2
        },
        {
          "text": "犹豫片刻，先在营外观察几日……",
          "effects": [
            { "type": "advance_time", "days": 5 },
            { "type": "set_flag", "flag": "ZHU_HESITATE" }
          ],
          "next_scene": 3
        }
      ]
    },
    {
      "scene_id": 2,
      "text": "郭子兴见你相貌不凡，谈吐间颇有胆识，当即收为亲兵，并赐名‘元璋’。",
      "effects": [
        { "type": "change_name", "value": "朱元璋" },
        { "type": "add_relation", "target": "GUO_ZIXING", "value": 20 }
      ]
    }
  ]
}
```

**字段说明**：
- `trigger`：触发条件块，支持时间、地点、身份、标记、概率等多维度组合。
- `scenes`：事件场景序列。一个事件可包含多个连续场景，通过`next_scene`跳转。
- `choices`：玩家选项。每个选项可携带多个`effects`（效果指令），执行后可能跳转到下一场景。
- `effects`：效果指令，包括：改变身份、势力、属性、技能、亲密度；添加卡片；设置/清除标记；推进时间；触发合战/个人战；增减金钱等。

### 八、事件系统实现建议（文字游戏）

**1. 事件调度器（EventScheduler）**

```python
class EventScheduler:
    def __init__(self):
        self.events = load_events_from_json()  # 所有事件定义
        self.triggered_events = set()           # 已触发的事件ID

    def check_and_trigger(self, world_state):
        candidates = []
        for event in self.events:
            if event.id not in self.triggered_events:
                if self.check_conditions(event.trigger, world_state):
                    candidates.append(event)
        if candidates:
            # 按优先级排序，取最高优先级事件执行
            selected = max(candidates, key=lambda e: e.priority)
            self.execute_event(selected, world_state)
            self.triggered_events.add(selected.id)
            return True
        return False
```

**2. 条件检测函数**

采用数据驱动，`trigger`中的每个字段对应一个检测函数。例如：
```python
def check_condition(cond, world_state):
    if "time" in cond:
        if not check_time(cond["time"], world_state.time): return False
    if "location" in cond:
        if world_state.player.location != cond["location"]: return False
    if "flags_required" in cond:
        for flag in cond["flags_required"]:
            if flag not in world_state.event_flags: return False
    # ... 其他条件
    return True
```

**3. 事件效果执行器**

每个`effect`指令对应一个执行函数，统一修改`world_state`和`player`数据：
```python
def apply_effect(effect, world_state):
    if effect["type"] == "add_card":
        acquire_card(effect["card_id"])
    elif effect["type"] == "change_identity":
        world_state.player.identity = effect["value"]
    # ... 更多效果类型
```

**4. 分支剧情与存档**

事件中的玩家选择会影响后续剧情分支，这些分支通过设置不同的`event_flag`实现。存档时，`event_flags`集合需完整保存，确保读档后事件链不混乱。

**5. 文本资源管理**

建议将事件文本与代码分离，单独存储在`event_texts.json`或CSV中，便于后期润色和本地化。文本中可使用占位符（如`{player_name}`）动态替换。


### 九、总结

《洪武立志传》的剧情与事件系统是游戏叙事张力的核心来源。通过**主线事件链**，玩家将身临其境地经历朱元璋从乞丐到皇帝的波澜壮阔；通过**丰富的支线与随机事件**，明朝的政治生态、市井生活、江湖风貌得以徐徐展开。

在技术实现上，事件系统采用**数据驱动 + 条件检测 + 效果指令**的架构，使得添加新事件只需编写JSON配置，无需修改核心代码——这对于单人开发者持续扩充游戏内容是极为友好的设计。

至此，游戏的八大核心系统（核心数据、卡片、时间移动、合战、个人战、主命、小游戏、社交经济、剧情事件）已全部阐述完毕。这些系统相互咬合，共同支撑起一个既有《太阁立志传》高自由度神韵，又充满明朝历史质感的文字游戏世界。
